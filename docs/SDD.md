Software Design Document (SDD)
Section 1 — System Overview
1.1 Purpose
The purpose of NairaLens AI is to provide users with a personalized financial intelligence platform that transforms financial records into actionable insights through artificial intelligence, continuous learning, and conversational interaction.
Unlike traditional finance applications, NairaLens does not simply display financial data. It interprets that data, builds long-term financial understanding, and helps users make better financial decisions over time.

1.2 Scope
Version 1 of the platform includes:
Secure user accounts
Bank statement upload
Transaction extraction
Financial analysis
Financial Intelligence Graph
AI-powered conversations
Personalized recommendations
Continuous Financial Learning
The platform is designed to evolve toward direct financial integrations while maintaining the same architectural foundation.

1.4 Core Design Principles
Every engineering decision should align with these principles.
Domain-Driven Design
Business domains are isolated to reduce coupling and improve maintainability.

AI-Native
Artificial intelligence is not an add-on feature.
It is a first-class component of the system architecture.

Modular Architecture
The platform uses a Modular Monolith that can evolve into microservices if future growth requires it.

Explainability
Every insight generated by the AI should be explainable through supporting financial evidence.

Continuous Learning
Each interaction should improve the platform's understanding of the user's financial life.

Privacy by Design
Financial data is treated as highly sensitive.
Security, encryption, and user control are fundamental design requirements.

Section 2 – Backend Architecture

2.1 Architectural Style
NairaLens will adopt a Modular Monolith Architecture based on Domain-Driven Design (DDD) principles.
Each business domain is logically separated into independent modules that encapsulate their own responsibilities, business logic, and data access.
This approach provides the simplicity required for rapid MVP development while preserving a clear path to future service decomposition if scaling demands it.

2.2 Architectural Principles
The backend follows these architectural rules.
Rule 1 — Domains Own Their Business Logic
Every domain owns its business rules.
Example:
Identity owns authentication.
Financial Data owns statement processing.
Financial Intelligence owns financial reasoning.
Conversation owns chat interactions.
No domain should implement another domain's business logic.

Rule 2 — Internal Communication Through Services
Domains communicate through well-defined service interfaces.
Avoid direct access to another domain's database models wherever possible.
For example:
✅ Good
financial_intelligence.analyze(statement_id)
❌ Bad
statement.transactions.update(...)
This reduces coupling and makes future refactoring much easier.

Rule 3 — Business Logic Never Lives in API Routes
FastAPI routes should remain thin.
Their responsibilities are limited to:
Validate requests
Authenticate users
Call application services
Return responses
All business logic belongs in the domain layer.

Rule 4 — AI Is a Consumer, Not an Owner
The AI does not own data.
Instead, it consumes information from:
Financial Data Domain
Financial Intelligence Domain
Conversation Domain
This ensures that replacing or upgrading the language model does not require changes to the core business logic.

Rule 5 — Background Jobs Handle Heavy Processing
Long-running operations should execute asynchronously.
Examples include:
Statement parsing
Transaction categorization
Insight generation
Recommendation generation
The API should acknowledge receipt quickly and allow background workers to complete processing.

2.6 Dependency Rules
To maintain a clean architecture, dependencies should only flow inward toward the business logic.
Frontend
    │
    ▼
API Layer
    │
    ▼
Application Services
    │
    ▼
Domain Services
    │
    ▼
Repositories
    │
    ▼
Database
A higher layer may depend on a lower layer, but not the reverse.
For example:
API routes can call services.
Services can call repositories.
Repositories access the database.
Repositories should never invoke API routes or UI code.

2.7 Asynchronous Processing Flow
Uploading a statement should not block the user while AI processing occurs.
The processing pipeline should follow this sequence:
User uploads PDF
        │
        ▼
Store File
        │
        ▼
Create Statement Record
        │
        ▼
Queue Background Job
        │
        ▼
Return "Processing" Response
        │
        ▼
Background Worker
        │
        ▼
Extract Transactions
        │
        ▼
Generate Financial Intelligence
        │
        ▼
Generate Financial Story
        │
        ▼
Mark Analysis Complete
This approach keeps the application responsive and prepares the architecture for larger workloads.

2.8 Error Handling Strategy
The platform should fail gracefully and provide actionable feedback.
Examples:
Scenario
Expected Behavior
Invalid PDF
Inform the user and reject the upload.
Unsupported bank format
Notify the user that the format is not yet supported.
AI processing failure
Retry automatically; if unsuccessful, present a clear error and allow reprocessing.
Database error
Log the error, return a generic message, and prevent data corruption.
